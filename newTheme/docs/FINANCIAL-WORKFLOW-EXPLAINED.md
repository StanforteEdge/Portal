# Complete Financial Workflow: Request to Retirement

## 🎯 **Understanding the Full Financial Cycle**

---

## 📊 **The Complete Financial Process**

### **Overview:**
```
Request → Approval → Payment Voucher → Payment → Receipt → Retirement → Audit
```

Let me break this down step by step:

---

## 🔄 **Stage 1: REQUEST (Asking for Money)**

### **What is it?**
A formal request to spend money. This is the **INTENTION** to spend.

### **Types:**
1. **Petty Cash Request** - Small amounts for immediate needs
2. **Expense Reimbursement** - Staff spent their own money, asking for refund
3. **Purchase Request** - Need to buy something
4. **Payment Request** - Need to pay a vendor/supplier

### **Example:**
```
John needs office supplies:
- Printer paper: ₦5,000
- Pens: ₦2,000
- Total: ₦7,000

He creates a REQUEST
Status: "Pending Approval"
```

### **Key Point:**
**No money has moved yet!** This is just asking permission.

---

## 🔄 **Stage 2: APPROVAL (Getting Permission)**

### **What happens?**
Request goes through approval chain:
```
Staff → Team Lead → Account Officer → COO → (ED if high value)
```

### **Each approver checks:**
- Is this necessary?
- Is budget available?
- Is amount reasonable?
- Is it aligned with strategy?

### **Example:**
```
John's request:
✓ Team Lead: "Yes, we need supplies"
✓ Account Officer: "Budget available"
✓ COO: "Approved"

Status: "Approved"
```

### **Key Point:**
**Still no money moved!** Just permission granted.

---

## 🔄 **Stage 3: PAYMENT VOUCHER (Authorization to Pay)**

### **What is it?**
A formal document authorizing the accountant to release money. This is the **EXECUTION** document.

### **When is it created?**
**After request is approved**, when ready to make payment.

### **Two Scenarios:**

#### **Scenario A: Internal Payment (Staff/Petty Cash)**
```
1. Request approved
2. Staff goes to accountant
3. Accountant creates Payment Voucher
4. Staff signs voucher
5. Accountant gives cash
6. Staff signs receipt
```

#### **Scenario B: External Payment (Vendor/Supplier)**
```
1. Purchase request approved
2. Vendor delivers goods/services
3. Vendor submits invoice
4. Accountant creates Payment Voucher
5. Payment made (bank transfer/cheque)
6. Vendor signs receipt
```

### **Example:**
```
John's approved request (₦7,000):

Payment Voucher #PV-00456
Date: 02 Dec 2024
Pay to: John Doe
Amount: ₦7,000
Purpose: Office supplies (per REQ-00123)
Approved by: COO

[Signature: Accountant]
[Signature: John (received)]
```

### **Key Point:**
**This is when money actually moves!** Payment Voucher = Cash out.

---

## 🔄 **Stage 4: PAYMENT (Money Moves)**

### **What happens?**
Actual transfer of money.

### **Methods:**
1. **Cash** - Physical money from petty cash box
2. **Bank Transfer** - Electronic payment
3. **Cheque** - Written cheque to vendor
4. **Mobile Money** - Digital payment

### **Example:**
```
Accountant:
1. Opens petty cash box
2. Counts ₦7,000
3. Gives to John
4. John signs Payment Voucher
5. Accountant files voucher
```

### **Key Point:**
**Money has left the organization!** Must be tracked.

---

## 🔄 **Stage 5: RECEIPT (Proof of Purchase)**

### **What is it?**
Proof that money was actually spent on what was approved.

### **Types:**
1. **Vendor Receipt** - From shop/supplier
2. **Invoice** - From service provider
3. **Bill** - From utility company
4. **Delivery Note** - Proof goods received

### **Example:**
```
John goes to shop:
1. Buys printer paper (₦5,000)
2. Buys pens (₦2,000)
3. Gets receipts from shop
4. Total: ₦7,000

Receipts show:
- Date of purchase
- Items bought
- Amounts paid
- Shop stamp/signature
```

### **Key Point:**
**Proof that money was used correctly!** No receipt = problem.

---

## 🔄 **Stage 6: RETIREMENT (Closing the Loop)**

### **What is it?**
Submitting receipts to prove money was spent correctly. This **CLOSES** the financial transaction.

### **What staff submits:**
1. Original receipts
2. Payment Voucher (signed)
3. Any change/balance
4. Retirement form

### **Example:**
```
John returns to accountant:
1. Submits receipts (₦7,000)
2. Receipts match approved items
3. Accountant verifies
4. Transaction marked "Retired"
5. File closed

If John spent ₦6,500:
- Returns ₦500 change
- Submits receipts for ₦6,500
- Accountant updates records
```

### **Key Point:**
**Transaction is complete!** Money accounted for.

---

## 🔄 **Stage 7: AUDIT (Verification)**

### **What is it?**
Periodic review to ensure all transactions are proper.

### **Auditor checks:**
1. Request was approved
2. Payment Voucher matches request
3. Receipts match payment voucher
4. All signatures present
5. No irregularities

### **Example:**
```
Auditor reviews John's transaction:
✓ Request REQ-00123 approved
✓ Payment Voucher PV-00456 issued
✓ Receipts total ₦7,000
✓ All signatures present
✓ Transaction retired

Status: "Clean"
```

## 🔗 **How Request and Payment Voucher Connect**

### **The Relationship:**
```
Request = "Can I spend money?"
Payment Voucher = "Here's the money"
```

### **They are LINKED but SEPARATE:**

#### **Request:**
- Created by staff
- Needs approval
- Shows intention
- Can be rejected
- No money moves

#### **Payment Voucher:**
- Created by accountant
- After approval
- Authorizes payment
- Money moves
- References request

### **Link:**
```
Payment Voucher #PV-00456
Purpose: Office supplies
Reference: Request #REQ-00123 ← LINK!
```

## 🎯 **Recommended System Design**

### **Database Structure:**

```sql
-- Main request
sta_request_instances
- id (REQ-00123)
- status (draft/submitted/approved/paid/retired)
- created_by
- total_amount

-- Payment voucher (linked to request)
sta_payment_vouchers
- id (PV-00456)
- request_id (REQ-00123) ← FOREIGN KEY
- voucher_number
- payee_name
- amount
- payment_method
- payment_date
- status (pending/paid)

-- Retirement (linked to request)
sta_retirements
- id
- request_id (REQ-00123) ← FOREIGN KEY
- payment_voucher_id (PV-00456) ← FOREIGN KEY
- retired_date
- retired_by
- total_receipts_amount
- balance_returned
- status (pending/completed)

-- Receipt attachments
sta_retirement_receipts
- id
- retirement_id
- file_id
- description
- amount
```

## 🔄 **Complete Workflow in System**

### **Step 1: Create Request**
```
User: /requests/create
Action: Fill form, submit
Database: INSERT into sta_request_instances
Status: "submitted"
```

### **Step 2: Approval**
```
Approvers: /requests/approvals
Action: Approve
Database: UPDATE sta_workflow_history
Status: "approved"
```

### **Step 3: Create Payment Voucher**
```
Accountant: /requests/{id} → Payment Voucher tab
Action: Click "Create Payment Voucher"
Database: INSERT into sta_payment_vouchers
         SET request_id = REQ-00123
Status: "approved" → "payment_pending"
```

### **Step 4: Make Payment**
```
Accountant: Payment Voucher tab
Action: Click "Mark as Paid"
Input: Payment method, date
Database: UPDATE sta_payment_vouchers
         SET status = 'paid'
         UPDATE sta_request_instances
         SET status = 'paid'
Status: "paid"
```

### **Step 5: Retirement**
```
User: /requests/{id} → Retirement tab
Action: Upload receipts
Database: INSERT into sta_retirements
         INSERT into sta_retirement_receipts
Status: "paid" → "pending_retirement"

Accountant: Verify receipts
Action: Click "Approve Retirement"
Database: UPDATE sta_retirements
         SET status = 'completed'
         UPDATE sta_request_instances
         SET status = 'retired'
Status: "retired"
```

## 📊 **Status Flow**

```
draft
  ↓
submitted (awaiting approval)
  ↓
approved (ready for payment)
  ↓
payment_pending (PV created)
  ↓
paid (money given)
  ↓
pending_retirement (awaiting receipts)
  ↓
retired (receipts submitted & verified)
  ↓
closed (transaction complete)
```

## 🎉 **Summary**

**Request and Payment Voucher are LINKED but SEPARATE:**

- **Request** = Asking permission (before money)
- **Payment Voucher** = Giving money (after approval)
- **Retirement** = Proving money used correctly (after spending)

**Best approach:**
- **One request page with 3 tabs**
- Tab 1: Request details
- Tab 2: Payment voucher (created after approval)
- Tab 3: Retirement (submitted after spending)

**This gives complete financial lifecycle in one place!** 🚀
